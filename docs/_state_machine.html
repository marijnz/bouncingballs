State Machines

Instantiation

There are basically two ways to create a state or a state machine: Use the low-level calls yourself, or use the functors State, StateMachine, and StateTransitions. Below you will see how to use the latter.

Creating a State in an existing State Machine

This can be used to hook into the main, built-in top-level state machine /game.

1 State{
2  -- The unqualified (local) name of the new state.
3  name = "myState",
4 
5  -- The fully qualified name of the parent state machine in which the state should be created
6  -- This is NOT optional here!
7  parent = "/game",
8 
9  -- This describes the listeners of the events of this new state.
10  -- This is optional and does not need to be specified.
11  eventListeners = {
12  enter = {
13  -- Here goes a list of anonymous functions
14  function(eventData) logMessage("Entered /game/myState") end,
15  function(eventData) logMessage("Yes, more than one listener is possible!") end
16  },
17  leave = {
18  function(eventData) logMessage("Left /game/myState") end
19  },
20  update = {
21  function(eventData) logMessage("Updating /game/myState") end
22  }
23  }
24 }

Adding transitions to an existing State Machine

The state transitions table must have a "parent" key but must be a pure array otherwise, i.e. it must only have consecutive integer keys (t[1], t[2], t[3], ...)

1 StateTransitions{
2  -- The fully qualified name of the parent state machine. The states used in the transitions must exist in this state machine.
3  parent = "/game",
4 
5  -- Will always enter 'myState' when the '/game' state machine is entered.
6  -- The condition is optional, in which case a return value of true is assumed.
7  { from = "__enter", to = "myState" },
8 
9  -- When the ESCAPE key was pressed, the '/game' state machine will be left.
10  { from = "myState", to = "__leave", condition = function() return InputHandler:wasTriggered(Key.Escape) end },
11 
12  -- { ... } and so on.
13 }

Creating a new State Machine

Below you will find two examples. The first shows how to create a new state machine that will be nested in a given parent. The second one shows how to create a top-level state machine (scroll down...)

1 -- Create a state machine "/game/nestedStateMachine"
2 StateMachine{
3  -- The unqualified (local) name for this new state machine.
4  -- If 'parent' is not specified, 'name' will become the qualified name
5  -- and this state machine will be created as a top-level state machine.
6  name = "nestedStateMachine",
7 
8  -- The fully qualified name of the parent state machine.
9  -- This is optional.
10  parent = "/game",
11 
12  -- Event listeners work the same with StateMachine as with State and are also optional.
13  eventListeners = { --[[ ... ]]-- },
14 
15  -- A list of inner states.
16  states = {
17  -- First inner state
18  {
19  -- Unqualified (local) name of the inner state.
20  name = "innerState",
21 
22  -- Note that you don't need to specify a parent here. If you do, you will get a warning.
23 
24  eventListeners = {
25  enter = { function(eventData) logMessage("Entered /game/nestedStateMachine/innerState") end }
26  }
27  },
28  -- Second inner state
29  {
30  -- Again, the unqualified (local) name of the inner state.
31  name = "anotherInnerState",
32 
33  -- Note now that the name is all that has to be specified for the simplest inner state.
34  },
35  -- And so on...
36  },
37 
38  -- A list of inner or rather nested state machines.
39  stateMachines = {
40  -- First nested state machine
41  {
42  -- This one is unqualified again
43  name = "innerMachine",
44 
45  -- No need to specify the parent here.
46 
47  -- Event listeners are optional...
48 
49  -- Here we go again...
50  states = {
51  {
52  name = "deepInside",
53  eventListeners = { enter = { function(eventData) logMessage("Entered /game/nestedStateMachine/innerMachine/deepInside") end } }
54  }
55  },
56 
57  transitions = { --[[ See the 'transitions' member below ]]-- }
58  },
59 
60  -- And so on...
61  },
62 
63  -- And finally, the transitions within this state machine.
64  -- Works just like using StateTransitions, except the 'parent' member does not need to be specified.
65  transitions = {
66  { from = "__enter", to = "innerState" },
67  { from = "innerState", to = "innerMachine" },
68  { from = "innerMachine", to = "__leave" },
69  }
70 }

Now we create a top-level state machine.

1 StateMachine{
2  name = "myTopLevel",
3  states = {
4  {
5  name = "level1",
6  eventListeners = {
7  update = {
8  function(eventData) logMessage("This will be shown many, many times!") end,
9  -- ...
10  }
11  }
12  }
13  },
14  transitions = {
15  { from = "__enter", to = "level1" }
16  -- Never leave level1 ...
17  }
18 }
