3 Game Objects {#GameObjects}
4 ============
5 
6 Each game object can have multiple components. A game object by itself has no capabilities other than positioning and orienting itself. In order to give a game object more capabilities, you can use components
7 
8 Creating a Game Object {#GameObjectCreate}
9 ----------------------
10 
11 To create a game object in Lua, you need to use the global `GameObjectManager` and its function `createGameObject`. Every game object needs a globally unique ID => GUID. It can be used to get a game object instance from the `GameObjectManager`. This GUID is given when creating the game object.
12 
13 ~~~{.lua}
14 local theGameObject = GameObjectManager:createGameObject("TheGuid")
15 ~~~
16 
17 In the example above, `"TheGuid"` is the GUID of the game object.

53 Components {#Components}
54 ==========
55 
56 Every component incorporates a certain capability, i.e. something that it can do. An example would be the `RenderComponent` which is capable of loading and rendering a given mesh. Below is a list of all currently available components and their functionality. But first we explain how to create a component in general.
57 
58 Creating A Component {#ComponentCreate}
59 --------------------
60 
61 A game object serves as the factory for components. From Lua, you can create a component using a game object like this:
62 
63 ~~~{.lua}
64 local theComponent = theGameObject:createXyzComponent(...)
65 ~~~
66 
67 As a concrete example, this is how you create a `ScriptComponent`:
68 
69 ~~~{.lua}
70 local sc = theGameObject:createScriptComponent()
71 sc:setPath("data/models/TheNameOfTheModel.thModel")
72 ~~~
73 
74 The component is then created with the game object as the sole owner of that component. _Components can not be shared between objects_.
75 
76 Component States {#ComponentStates}
77 ----------------
78 
79 The following describes which states exist:
80 
81 | Value | Can be Set by User |
82 | ----- | ------------------ |
83 | `ComponentState.Initial` | no |
84 | `ComponentState.Active` | yes |
85 | `ComponentState.Inactive` | yes |
86 
87 By default, every component is set to `ComponentState.Active` after it has been initialized. Before that, it is in the state `ComponentState.Initial`.
88 
89 To change the state of an individual component, you can call the function `setState` with either of the arguments given above which say "yes, can be set by user". If you want to change the state of all components that belong to a game object, you can call `setComponentStats` on that game object.
90 
91 Render Component {#RenderComponent}
92 ----------------
93 
94 Used to render a given mesh at the transformation of the owning game object. Use it like this:
95 
96 ~~~{.lua}
97 local rc = theGameObject:createRenderComponent()
98 rc:setPath("data/models/theNameOfTheModelFile.thModel")
99 local thePath = rc:getPath() -- Returns the argument given above
100 ~~~
101 
102 Unlike any other object in the game, a render component supports scaling. You can manipulate the scale of the rendered object using `setScale`:
103 
104 ~~~{.lua}
105 rc:setScale(rc:getScale():mulScalar(2.0)) -- Scale it to twice the original size
106 rc:setScale(Vec3(2.0, 1.0, 1.0)) -- Distorted in X direction
107 ~~~

114 Physics Component {#PhysicsComponent}
115 -----------------
116 
117 The physics component is used to create a rigid body which represents the game object in the physics world of the game. To create a rigid body, you have to use the construction info `RigidBodyCInfo` to define the properties of your rigid body:
118 
119 ~~~{.lua}
120 local pc = theGameObject:createPhysicsComponent()
121 local cinfo = RigidBodyCInfo()
122 cinfo.motionType = MotionType.Fixed
123 cinfo.mass = 42
124 cinfo.shape = PhysicsFactory:createSphere(1.337)
125 cinfo.friction 0.33
126 -- ...
127 local rb = pc:createRigidBody(cinfo)
128 ~~~
129 
130 See the [Physics](\ref Physics) page for more information.

132 Script Component {#ScriptComponent}
133 ----------------
134 
135 Used to update a certain game object. A script component has three 'slots': initialize, update, destroy. You can register any function with the script component for any of the given slots, provided your function has the correct signature.
136 
137 | Function Slot | Signature |
138 | ------------- | --------- |
139 | initialize | `function(gameObjectGuid)` |
140 | destroy | `function(gameObjectGuid)` |
141 | update | `function(gameObjectGuid, elapsedSeconds)` |
142 
143 The syntax for registering those functions is simple:
144 
145 ~~~{.lua}
146 function initialize(guid) print("Initializing " .. guid)
147 function destroy(guid) print("destroying " .. guid)
148 function update(guid) print("update " .. guid)
149 ~~~
150 
151 The most useful slot is the update slot. Since initialization happens only when There can only be exactly one function active for each of the slots
152 
153 You can control whether the slots are called by setting the component state. See [Component States](\ref ComponentStates) for more details.
